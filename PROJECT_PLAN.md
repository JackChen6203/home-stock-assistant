# 家庭（個人）庫存系統 MVP 專案規劃

## 1. 產品目標
建立一個「可免費運作」的家庭/個人庫存管理 MVP，讓使用者能：
1. 透過 Siri / Google 助理快速新增「需要補貨的生活用品與食物」。
2. 透過 GPS 地理圍欄，在接近賣場時收到購物提醒。
3. 串接台灣電子發票載具（涵蓋常見載具方式），當偵測到購買品項時自動扣除待購清單，也可手動刪除。
4. 在維持低成本與高可行性的前提下，先驗證核心價值。

---

## 2. 使用者故事（MVP 範圍）

### US-01 語音新增
- **身為** 使用者
- **我希望** 對 Siri 或 Google 助理說「幫我記得買牛奶/衛生紙」
- **以便** 不需打開 App 也能新增待購項目。

### US-02 到店提醒
- **身為** 使用者
- **我希望** 當我到住家附近常去賣場（全聯、家樂福等）時收到通知
- **以便** 不漏買清單中的物品。

### US-03 發票自動扣除
- **身為** 使用者
- **我希望** 載具發票出現「牛奶」時自動將牛奶從待購清單扣除
- **以便** 減少手動維護。

### US-04 手動調整
- **身為** 使用者
- **我希望** 可以手動刪除/更正清單項目
- **以便** 避免辨識錯誤造成清單不準。

---

## 3. 系統功能拆分

## 3.1 語音輸入模組

### Siri（iOS）
- 使用 **Apple Shortcuts + App Intent（或 URL Scheme）**。
- 流程：
  1. 使用者語音觸發捷徑。
  2. 捷徑將「品項文字」送到後端 API。
  3. 後端做標準化（同義詞、單位）後寫入待購清單。

### Google 助理（Android）
- 使用 **Google Assistant + App Actions / Shortcut**。
- 流程同上，提交 `POST /items/add-by-voice`。

### MVP 取捨
- 第一階段不做複雜語意理解，只做關鍵詞抽取與品項映射字典。

## 3.2 地理圍欄提醒模組
- 每位使用者可設定 3–10 個常去賣場。
- 行動端建立 Geofence（半徑 150m~300m）。
- 進入圍欄時觸發本機通知：
  - 「你附近有賣場，待購：牛奶、衛生紙…」。
- 若手機離線，仍可透過本機快取清單通知。

## 3.3 台灣電子發票載具整合模組

### 載具類型（設計需全覆蓋）
1. **手機條碼載具（/ABC+123）**
2. **自然人憑證條碼載具**
3. **悠遊卡（EasyCard）**
4. **一卡通（iPASS）**
5. **icash 2.0**
6. **信用卡載具（由發卡機構/平台綁定）**
7. **跨境電商電子郵件載具**

> 備註：不同載具的可串接程度與授權流程差異大，MVP 建議先以「手機條碼載具」做完整自動化，其他載具先提供資料匯入或半自動方式。

### 發票資料來源策略（MVP）
- 優先：使用者授權後定期同步可取得之發票明細。
- 備援：支援手動匯入 CSV/截圖 OCR（若官方/第三方 API 限制）。

### 自動扣除邏輯
1. 發票品名清洗（全形半形、品牌前綴去除、規格/容量詞去除）。
2. 多層品項比對（避免只靠完全 match）：
   - **同義詞字典比對**：例如「阿猴鮮奶」「鮮乳」「保久乳」→「牛奶」。
   - **品牌 + 品類拆解比對**：先去品牌詞，再比對品類詞（如「XX農場鮮奶」→「鮮奶」→「牛奶」）。
   - **籠統類別映射**：例如發票出現「青菜」「蔬菜」可對應清單的「葉菜類/高麗菜」群組；「水果」可對應「香蕉/蘋果」群組。
   - **模糊比對**（Levenshtein / 關鍵詞 / N-gram）：處理錯字與簡寫。
3. 類別到具體品項的扣除策略：
   - 若待購清單有「泛類別」（如「水果 x2」），發票命中同類即直接扣除。
   - 若待購清單是「具體品項」（如「蘋果」）但發票僅寫「水果」，先標記為「候選扣除」待使用者確認。
   - 若同時存在多個候選（如蘋果、香蕉都未買），依優先級/建立時間推薦扣除目標。
4. 成功匹配後：
   - 待購數量 -1（或直接標記已購）。
   - 寫入「自動扣除紀錄」與「比對依據」（命中詞、分數、規則）。
5. 信心不足（低於閾值）時：
   - 進入「待確認」列表，由使用者一鍵確認。

## 3.4 清單管理模組
- 類別：食物 / 日用品 / 其他。
- 欄位：名稱、數量、優先級、建立時間、最後更新時間。
- 操作：新增、刪除、標記已購、復原。

## 4. MVP 技術架構（免費優先）

## 4.1 建議技術棧
- **前端（行動端）**：Flutter（iOS/Android 單一程式碼）
- **後端 API**：FastAPI（Python）
- **資料庫**：Supabase Postgres（免費層）
- **認證**：Supabase Auth（Email/Google/Apple）
- **排程任務**：GitHub Actions Cron / Supabase Edge Functions（擇一）
- **通知**：Firebase Cloud Messaging（免費）
- **地圖/GPS**：手機原生定位 + Geofence API

## 4.2 邏輯分層
- `mobile-app`：UI、語音入口、地理圍欄、推播接收
- `api-server`：清單管理、比對引擎、載具同步排程
- `db`：users/items/receipts/deductions/audit_logs

## 4.3 成本估算（MVP）
- 雲端成本：可控制在 **0 元 ~ 極低費用**（依免費額度）。
- 必要條件：
  - 低 DAU（例如 < 1000）
  - 排程頻率保守（如每 3~6 小時同步一次發票）

## 5. 資料模型（簡版）

- `users`
  - id, email, locale, created_at

- `shopping_items`
  - id, user_id, canonical_name, display_name, qty_needed, status, priority, source(voice/manual), created_at

- `receipt_entries`
  - id, user_id, carrier_type, invoice_no, invoice_date, raw_item_name, normalized_name, amount

- `deduction_logs`
  - id, user_id, shopping_item_id, receipt_entry_id, match_score, mode(auto/manual), created_at

- `carrier_accounts`
  - id, user_id, carrier_type, masked_identifier, sync_mode(api/import), last_sync_at

---

## 6. API 草案
- `POST /items`：手動新增項目
- `POST /items/add-by-voice`：語音入口新增
- `DELETE /items/{id}`：手動刪除
- `POST /geofence/events`：行動端上報進入賣場事件
- `POST /carrier/sync`：觸發載具同步
- `POST /carrier/import`：匯入發票資料
- `POST /deductions/reconcile`：執行自動扣除比對
- `GET /items/pending`：取得待購項目

## 7. 開發里程碑（8 週）

### Phase 1（第 1–2 週）
- 建立資料庫、帳號系統、基本 CRUD。
- 完成手動新增/刪除清單。

### Phase 2（第 3–4 週）
- 串 Siri / Google 助理入口。
- 完成語音新增 API + 品項正規化。

### Phase 3（第 5–6 週）
- 完成地理圍欄與到店通知。
- 加入待購快照快取機制。

### Phase 4（第 7–8 週）
- 完成手機條碼載具同步與自動扣除。
- 建立「待確認比對」流程與操作頁。

---

## 8. 風險與對策
1. **載具 API/資料可得性不一**
   - 對策：先完成手機條碼載具；其他載具先支援匯入流程。
2. **發票品名辨識不穩定**
   - 對策：建立品牌別名字典 + 人工確認機制。
3. **背景定位耗電與系統限制**
   - 對策：限制圍欄數量、調整半徑、用系統原生事件而非高頻定位。
4. **免費額度超限**
   - 對策：降低同步頻率、縮減紀錄保存時間、啟用批次處理。

---

## 9. 成功指標（MVP KPI）
- 每週至少 3 次語音新增使用率 > 40%。
- 到店提醒點開率 > 25%。
- 自動扣除正確率 > 75%。
- 手動修正率逐月下降。

---

## 10. 下一步（建議立刻執行）
1. 先做「手機條碼載具 + 手動清單 + 到店提醒」最小閉環。
2. 蒐集 20–50 位早期使用者發票品名資料，優化比對字典。
3. 觀察 2 週後再決定是否擴充其他載具的「全自動」串接。
